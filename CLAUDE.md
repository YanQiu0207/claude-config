- 除非用户在当前会话中明确要求改用其他称呼

## 沟通方式

- 使用中文回复

## 沟通要求

- **事实性内容必须有来源支撑**：回答或写入任何事实性内容（工具行为、API、框架特性、技术细节等）时，必须有可靠来源（官方文档、WebFetch 确认、实测结果）；若不确定，明确说明"不确定"或"需要确认"，不能凭推断或训练数据中的模糊印象给出确定性的陈述。

## 任务分流

除非项目已有更严格且明确的规范，否则按本文件执行。

- **一次性小修、单文件改动**：按下面"工程强制规则"与"协作底线"执行即可，不走阶段制。
- **多文件、多阶段、需多 agent 并行、或需本地台账跟踪的复杂任务**：按 `~/.ai-rules/agent-delivery-workflow.md` 执行。阶段制、双评审、worker 巡检节奏、agent 资源释放、台账模板均在其中。

## 工程强制规则

完整规范见 `~/.ai-rules/engineering-standards.md`，核心要点：

- 所有关键路径必须有完善日志，覆盖关键输入、关键输出、异常、重试、降级和耗时统计。
- 默认优先使用结构化日志；跨模块和跨项目尽量统一字段名，例如 `event`、`trace_id`、`request_id`、`module`、`status`、`elapsed_ms`。
- 日志时间格式统一为 `2026-04-15 13:46:03.836`，精确到毫秒。
- 涉及第三方库或第三方服务时，必须设计防腐层；业务逻辑不得直接耦合厂商 SDK、HTTP 细节或响应模型。
- 可调参数不得随意写死在代码中；凡是可能因环境、业务策略或性能调优而变化的值，优先放入配置。
- 新增功能时，优先同时补齐日志点、配置项和防腐层，不接受"先直连、后补抽象"的默认做法。
- 记录请求与响应时，必须避免泄露密码、密钥、令牌、Cookie、Authorization 头等敏感信息；必要时只记录摘要、长度、状态和脱敏字段。
- 若项目已有现成日志、配置或集成抽象方案，应优先复用并在其基础上扩展，不另起一套平行体系。

## 协作底线

- 独立子任务可以并行，但必须先锁定写入边界。
- 多 agent 复杂任务中，主会话默认优先负责理解用户新指令、重排计划和协调分工；能下发给 worker 的实现工作，尽量不要长期压在主会话自己身上。
- 用户中途追加指令时，默认先判断哪些 worker 可继续、哪些需要纠偏；不要因为收到新指令就让整体开发停摆。
- 进度表述必须和实际动作匹配：
  - 只做了阅读、查看、核对时，只能说"已查看 / 已核对 / 正在准备"
  - 只有在已创建任务、已更新阶段状态、已分派 agent、已开始改代码、或已运行脚手架/验证命令后，才能说"已开始 / 已启动 / 正在开发"
  - 禁止把"看一眼目录或文档"表述成"已经开始做下一阶段"
- 完成或不再需要的 agent 立即关闭，不长期占线程。
- 阶段或子任务收口后默认自动推进到下一动作，不等待用户再次催促；确实被外部阻塞才停下并说明原因。

## 写代码前的确认流程

- 任何涉及写代码的任务，无论大小，必须先与用户确认以下三点，收到明确同意后再开始编码：
  - 设计方案（技术选型、架构设计、模块划分等）
  - 开发计划（任务拆分、优先级、里程碑等）
  - 参与人数（影响任务分工与并行策略）
