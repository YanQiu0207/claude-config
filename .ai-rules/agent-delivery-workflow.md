# 代理开发工作流与 Markdown 模板

用于复杂项目：统一多 agent 并行、阶段闸门、评审闭环和本地 Markdown 台账。

## 1. 何时使用这套工作流

满足任一条件时启用：

- 多文件或多阶段开发
- 需要多个 agent 并行
- 需要本地台账跟踪进度、评审、bug 和修复
- 需要明确“先评审，修完再进下一阶段”

## 2. 协作方式

### 2.1 主会话

主会话负责拆阶段、划并行边界、分派 agent、集成代码、做阶段闸门判断、维护 md 台账。

默认原则：

- 能稳定下发给 worker 的实现工作，主会话优先做协调和集成
- 用户可随时追加要求，但主线不应停摆

### 2.1.1 用户中途追加指令时的默认处理

用户中途追加指令时，先判断它影响计划、worker 边界、阶段闸门还是关键路径。

- 方向仍正确：不中断，只更新计划、台账和集成口径。
- 需要纠偏但不必立刻打断：追加约束。
- 与新要求直接冲突，或会写坏关键路径：立即中断。
- 处理完后立即恢复并行推进、本地验证和台账同步。

禁止行为：

- 因新指令让整体开发停摆
- 可下发的实现长期压在主会话自己身上
- 只口头确认，不更新计划或转达约束

### 2.1.2 进度表述校准规则

进度口径：

- `已查看 / 已核对 / 正在准备`：仅用于阅读、核对、形成分工。
- `已开始 / 已启动 / 正在开发`：仅用于已分派 worker、已改代码，或已执行构建/测试等开发动作。
- `已完成`：仅用于结果已落地且已核对。

禁止表述：

- 把“看过”说成“已开始”
- 把“想过”说成“在推进实现”
- 在没有分工、代码动作或开发命令时，使用“已启动阶段开发”

### 2.1.3 Worker 巡检与接管

主会话必须定时巡检并行 worker，不得长期放任不看。

- worker 完成后立即集成或关闭。
- worker 长时间无结果、明显卡住、前提错误或输出不可用时，立即接管。
- 巡检必须有可见动作。
- 连续两次巡检都没有有效结果时，必须明确选择：继续等待、发送纠偏或直接接管。
- 接管后记录原因，并由主会话继续完成关键路径。

### 2.2 子 agent

子 agent 只在明确范围内并行开发或独立审核，不得扩范围，也不得决定阶段放行。

## 3. 先规划：落地到 `dev-plan/`

复杂功能先规划，再执行。规划统一落到 `dev-plan/`，用于记录目标、范围、方案、阶段拆分和风险；`dev-plan/` 只作为执行输入，不替代 `dev-run/`。

## 4. 先规划阶段，再搭执行台账，再按阶段执行

### 4.1 阶段规划

先规划一共有多少阶段，再进入开发。每个阶段至少明确：

- 阶段名
- 目标
- 交付物
- 并行分工
- 验收标准
- review 风险

阶段状态流转默认如下：

`未开始 -> 开发 -> 评审 -> 修复 -> 通过`

补充规则：

- 当前阶段未通过前，不进入下一阶段。
- `开发` 结束后必须进入 `评审`。
- `评审` 有 accepted finding 时，必须进入 `修复`。
- `修复` 完成并验证通过后，才能进入 `通过`。
- 若被外部条件阻塞，可临时标记 `阻塞`，但必须记录阻塞原因、责任人、下一动作和恢复条件。

### 4.2 目录建议

复杂项目建议目录：

```text
<project>/
  dev-plan/
    master-plan.md
    scope.md
    architecture.md
    stage-plan.md
    risk-register.md
  docs/
  app/ 或 src/
  frontend/        # 如果有前端
  data/            # 如果有本地状态或缓存
  dev-run/
    stage-status.md
    progress.md
    agent-board.md   # 仅并行协作时需要
    issue-log.md     # 进入评审或修复时建立
```

文档、评审与台账不要和代码混放。

各 Markdown 模板见 `~/.ai-rules/rule-appendix.md` §2.1-§2.5。

### 4.3 `dev-run/` 文档

`dev-run/` 是执行层台账，回答“当前做到哪里、下一步做什么、问题是否收口”。

核心文件：

- `stage-status.md`：阶段状态和放行结果。
- `progress.md`：执行流水、证据、下一步。

按需文件：

- `agent-board.md`：仅在有并行 agent 或需明确写入边界时建立；由主会话维护，记录各 agent 在做什么、当前状态、是否完成。
- `issue-log.md`：进入评审、发现 accepted finding、进入修复后建立；统一记录评审结论、问题状态、修复动作和验证结果。

默认真相源：

- 阶段状态只认 `dev-run/stage-status.md`
- 当前动作与下一步只认 `dev-run/progress.md`
- 并行协作时，ownership 与写入边界只认 `dev-run/agent-board.md`
- 问题是否收口只认 `dev-run/issue-log.md`

### 4.4 每个阶段必须使用的 md

- `开发`：
  - `stage-status.md`：阶段状态
  - `progress.md`：执行记录、证据、下一步
  - `agent-board.md`：仅在并行协作时使用
- `评审`：
  - `stage-status.md`：状态切到 `评审`
  - `progress.md`：开发完成证据、评审启动动作
  - `issue-log.md`：评审意见、accepted finding、放行结论
  - `agent-board.md`：仅在 Review A/B 并行时使用
- `修复`：
  - `stage-status.md`：状态切到 `修复`
  - `progress.md`：修复动作和重跑验证
  - `issue-log.md`：问题状态、修复内容、验证结果、收口结论
  - `agent-board.md`：仅在修复任务并行时使用
- `通过`：
  - `stage-status.md`：状态切到 `通过`
  - `progress.md`：放行记录和下一阶段起始动作
  - `issue-log.md`：问题已收口且允许放行
  - `agent-board.md`：若存在，则在阶段通过前将各任务状态收口到最终态

### 4.5 评审、修复与放行

每个阶段开发完成后，默认执行：

1. 主会话先集成并做本地验证。
2. 冻结当前阶段范围。
3. 拉 2 个独立 agent 审核。
4. Review A 看协议、边界、数据语义、架构一致性。
5. Review B 看实现质量、回归风险、测试缺口、运行时问题。
6. accepted finding、修复动作和验证结果统一写入 `issue-log.md`。
7. 修复验证完成后，主会话才能把 `stage-status.md` 标为 `通过`。

### 4.6 本地验证失败后的默认续跑规则

本地验证失败后，不得停在失败输出；要么修复并重跑，要么记录阻塞并写回台账。

### 4.7 Agent 资源释放规则

agent 完成回传或不再需要后应尽快关闭，不得长期保活；若存在 `agent-board.md`，同步更新其最终状态。

## 5. 跨会话续跑

### 5.1 续跑三步

新会话先执行“续跑三步”，再写代码：

1. 读 `stage-status.md`，定位“未通过且最靠前”的阶段。
2. 读 `progress.md` 末尾 50~120 行，确认上次停点与下一动作。
3. 交叉核对 `issue-log.md`，并在存在 `agent-board.md` 时确认是否有未收口任务或 finding。

续跑规则：

- 若有未完成任务或未收口 finding，先继续该任务，不得跳到后续阶段。
- 若当前阶段为 `评审` 或“开发完成待评审”，首个执行动作必须是启动或补齐 Review A/B。
- 若台账冲突，先修一致性再继续实现。
- 续跑起步必须在 `progress.md` 追加会话恢复记录。

## 6. 统一质量底线

- 包含前端：默认目标是漂亮、专业、产品化；不接受简陋工具页风格；实时观测类界面默认支持自动更新。
- 包含多 agent 并行或复杂实现：按 §3、§4、§5 执行，不另起简化流程。
