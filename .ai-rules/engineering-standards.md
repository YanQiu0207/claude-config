# 全局工程规范

## 1. 日志

- 关键路径必须记录：开始、关键输入/输出摘要、异常、重试、降级、结束和耗时；复杂流程还要记录关键子步骤耗时。
- 第三方服务、数据库、缓存、文件系统、消息队列等关键依赖，必须记录调用前后摘要和耗时。
- 异常日志至少包含：异常类型、异常信息、触发动作、关键上下文、是否可重试、耗时；必要时附堆栈。
- 日志时间格式统一为 `yyyy-MM-dd HH:mm:ss.SSS`，示例：`2026-04-15 13:46:03.836`。
- 默认优先使用结构化日志；至少包含事件名、模块名、状态、请求/任务标识、耗时，并尽量统一字段名，如 `event`、`trace_id`、`request_id`、`module`、`status`、`elapsed_ms`。
- 请求和响应日志遵循最小必要原则，只记录摘要、白名单字段、截断内容或脱敏结果；禁止记录密码、Token、API Key、Cookie、Authorization 头和其他敏感原文。

## 2. 第三方依赖防腐层

- 第三方库和第三方服务必须通过防腐层接入；业务层不得直接依赖厂商 SDK、HTTP 细节、厂商响应模型或厂商异常类型。
- 防腐层负责请求封装、响应转换、错误映射、重试/超时、认证细节以及日志和耗时埋点；业务层只依赖内部接口、内部模型和内部错误语义。
- 若项目已有 `client`、`gateway`、`adapter` 等边界层，应优先复用，不重复造轮子。

## 3. 配置外置化

- 可能因环境、业务策略、性能调优或供应商切换而变化的值，不要写死在代码中。
- 地址、超时、重试、并发数、批量大小、阈值、开关项、缓存 TTL、日志级别、采样率、模型名、文件路径等原则上都应配置化。
- 只有真正稳定的协议常量或枚举值才适合保留在代码中；缺失关键配置时应尽早失败，敏感配置不得明文写入仓库。

## 4. 可读性与注释

- 特殊分支必须补充注释，说明触发条件、设计原因以及与常规路径的差异；不能让读代码的人只能靠猜测理解“为什么这里单独处理”。
- 复杂算法、关键状态机、非直观的边界处理和性能优化路径，必须在实现附近写明核心思路、关键约束以及为什么这样实现。
- 注释要解释“为什么”和“约束是什么”，不要把代码字面意思机械重复一遍。

## 5. 测试要求

### 5.1 覆盖底线

- 每个开发阶段必须包含测试，测试用例能跑通 + 关键路径覆盖后，才能进入评审。
- 接口层、核心业务逻辑、驱动/适配器必须有测试；无强制覆盖率数字。
- worker 实现任务时，**测试与实现代码同批交付**，不允许先交付实现再「后补」。
- Review B 必须检查测试缺口，测试覆盖严重不足视为阻塞项。

### 5.2 默认测试栈

- 后端：pytest + pytest-asyncio + httpx AsyncClient；集成测试用内存 SQLite，不依赖真实 DB 环境。
- 前端：复杂交互逻辑需补单元测试；纯展示组件可免。

### 5.3 测试用例变更纪律

- 测试用例写好后，原则上不允许随实现反复改动，不能通过改测试来迎合当前结果。
- 如确需修改测试用例，必须先有充分、可复核的理由，能说明原用例存在错误、歧义，或与真实目标不一致。
- 任何测试用例改动都必须记录变更原因、影响范围和判断依据，便于后续追溯。

## 6. 构建、编译与容器重启判定

### 6.1 基本原则

- 先判断当前交付物是否已经进入「运行中的进程 / 容器 / 静态产物」；只有运行时实际依赖的文件或镜像发生变化，才需要编译、重启或重建。
- `docker compose restart` 只会重启现有容器，不会重新构建镜像，也不会把工作区里的最新源码自动打包进镜像。
- `docker compose up -d --build <service>` 才会基于当前源码重建指定服务镜像并重新创建容器；当镜像内产物依赖源码构建结果时，默认使用这个命令，而不是只做 restart。
- 若项目启用了源码挂载、热更新或开发服务器，优先遵循项目现有运行方式；不要机械地每改一处都重建容器。

### 6.2 必须重新构建镜像并重启容器的场景

- 修改了会被打包进镜像的前端源码，且当前线上/本地访问的是容器内静态产物，而不是热更新开发服务器。
- 修改了 Dockerfile、`dockerfile_inline`、镜像构建阶段脚本、构建参数、基础镜像或安装依赖步骤。
- 修改了前端构建产物依赖的文件，如 `package.json`、锁文件、构建配置、Tailwind/Vite/Webpack 配置、静态资源处理逻辑等，且这些内容是在镜像构建阶段生效。
- 修改了后端 Python 代码，但当前容器不是源码挂载运行，而是镜像内复制代码后直接启动应用。
- 修改了运行镜像内才存在的模板、静态文件、Nginx 配置、Supervisor 配置、入口脚本等。

### 6.3 只需重启容器、不必重建镜像的场景

- 修改的是通过挂载卷注入容器的配置文件、脚本或数据文件，且容器重启后会重新读取这些文件。
- 修改的是环境变量、`.env`、`docker-compose.yml` 中的 `environment`、`command`、端口映射、挂载目录等运行参数，且镜像本身未变化。
- 服务只是因为临时状态异常、连接异常、进程卡住需要重启，而不是为了让新代码生效。

### 6.4 只需重新编译或重新运行进程、不必动容器的场景

- 本地直接运行前端构建命令或开发服务器，访问的也是本机进程，而不是容器内静态站点。
- 本地直接运行 Python、Node、Go 等开发进程，修改代码后只需按项目既有方式热更新、重启本地进程或重新执行构建。
- 修改的是测试代码、文档、台账、设计稿、截图、说明文档等，不影响正在提供服务的运行时。

### 6.5 前端静态站点的默认判定

- 若浏览器访问的是 Nginx / Caddy / Apache 容器内托管的前端静态文件，则前端源码改动后的默认动作是：`重新构建前端镜像 + 重建前端容器`。
- 仅执行 `docker compose restart frontend` 不足以让新的 JS/CSS 生效，除非当前容器本来就是源码挂载并在容器内实时构建。
- 遇到「源码已改、`pnpm build` 已通过、浏览器仍是旧页面」时，优先怀疑正在运行的仍是旧镜像，其次再查浏览器缓存。

### 6.6 后端服务的默认判定

- 若后端容器通过 `COPY . .` 或等价方式把源码打进镜像，则 Python/Node/Go 等后端源码变更后的默认动作是：`重新构建后端镜像 + 重建后端容器`。
- 若后端容器通过 bind mount 挂载源码并启用了自动重载，则以项目约定为准：可能只需要保存文件、重启进程或手动 reload，不必重建镜像。
- 若无法确定容器是「镜像内代码」还是「挂载代码」，必须先核对 Dockerfile / compose 配置，再决定是否重建。

### 6.7 执行要求

- 交付前必须明确告知用户：本次改动是否已经进入正在运行的容器；若没有，必须说明还差「重建镜像」「重启容器」还是「仅刷新页面」。
- 当需求目标包含「用户马上在浏览器看到新页面 / 新接口行为」时，不能只完成本地代码修改；必须把运行态也切到最新版本，或明确告知尚未完成这一步。
- 若仅做了 `restart`，不得口头等同于「已部署最新代码」；只有确认新构建产物或新镜像已实际被服务使用，才能这样表述。

## 7. 开发语言选型

### 7.1 后端

- **固定 Python**（硬约束）。默认栈：FastAPI + SQLAlchemy 2.0 + Alembic + Pydantic v2 + Uvicorn；异步用 asyncio，结构化日志用 structlog。
- 不得以「团队更熟悉」或「生态更大」为由切换到 Java / Go / Node 等其他语言；若确需偏离，先与用户明确确认并在项目文档中记录原因。

### 7.2 前端

- **无框架偏好**，按项目场景选最合适方案（体量、生态成熟度、UI 需求复杂度、后续维护人选均作为评估维度）。
- 涉及前端技术选型时，先给出至少两个候选方案的横向对比，由用户拍板，不得自作主张直接落地某一方案。

### 7.3 前端质量底线（与具体框架无关，不可降级）

- 界面必须高级、美观、产品化，不接受简陋工具页风格。
- UI 组件库、图标、动效、数据层的具体选型在所选框架生态内补齐。
- 前端布局和样式：需优先参考同主题的主流网页或成熟产品的常见做法，保持布局、信息层级、视觉节奏与交互模式符合该主题下的主流预期；不得闭门造车。

## 9. 换行符与编码

- 项目统一使用 LF 换行符；Windows 环境下 Git 全局设置 `core.autocrlf = input`，确保提交时 CRLF 自动转为 LF。
- AI 编码工具（Claude Code、Codex 等）生成或修改文件时，必须使用 LF 换行符，不得引入 CRLF。
